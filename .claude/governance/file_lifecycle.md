# File Lifecycle Policy

**Status:** Active governance rule — enforced from Phase 15 onwards.
**Authority:** Team Lead + Systemic Critic (L4 escalation required to override)

---

## The Rule

**A file that has lost its context to the system must not exist.**

A file loses its context when any of the following is true:

1. **Replaced** — another file now serves its purpose and the old file has not been explicitly converted to a pointer stub pointing to the replacement
2. **Superseded** — the file was a draft, a migration artifact, or a plan that has been fully executed; it retains no future reference value
3. **Orphaned** — no active agent, governance document, CLAUDE.md, or codebase file references it by path
4. **Duplicate** — its content is a verbatim or near-verbatim copy of another file with no distinct value added
5. **Dead pointer** — the file exists only to point to another path, and that target path no longer exists

---

## Mandatory Actions

### When replacing a file

The old file MUST be handled by one of these two options, in priority order:

1. **Delete it** — if no historical reference value exists
2. **Convert to pointer stub** — if it is referenced by external links or human memory; the stub contains only:
   ```
   > SUPERSEDED: This file has been replaced by [canonical file](../path/to/canonical.md).
   > Do not edit this file. All future changes go to the canonical file.
   ```
   Pointer stubs are themselves subject to deletion once no active reference to them remains.

There is no third option. A file cannot silently coexist with its replacement.

### When archiving completed work

- MVP plans that are **Complete** remain in `mvps/` as historical record — they are active references for the roadmap
- Governance files for **retired policies** must have a `> RETIRED:` header and be moved to `.claude/governance/retired/`
- Transient output directories (`validation_runs/`, `validation_runs_redacted/`) must never be committed; `.gitignore` must cover them

### When discovering an orphaned file

Any agent that discovers a file with no traceable purpose during their work MUST:
1. Flag it in their work output (do not silently skip it)
2. Escalate to Team Lead via the `dead_file_scan` skill output or a task note
3. Wait for deletion confirmation before proceeding with the blocked work

---

## What Is NOT Orphaned

The following categories are always considered active, regardless of apparent lack of references:

- `CLAUDE.md` — always active (session entrypoint)
- `.claude/governance/*.md` — active unless explicitly `> RETIRED:`
- `mvps/MVP_Phase*.md` — active historical record of completed and planned phases
- `roadmaps/Roadmap_*.md` — active planning artifact
- `tests/*.py` — active until explicitly removed by a test cleanup phase
- `claudeclockwork/**/*.py` — active unless `__pycache__` or `*.pyc`
- `registry/`, `configs/`, `.claude/config/` — active configuration files

---

## Detection: `dead_file_scan` Skill

The `dead_file_scan` skill (Phase 15, `meta` category) is the canonical automated tool for detecting orphaned files. Run it before any new phase begins:

```bash
python3 -m claudeclockwork.cli --skill-id dead_file_scan --inputs '{"dry_run": true}'
```

High-confidence candidates must be reviewed and resolved before the phase work begins.

---

## Enforcement

- **Pre-phase check:** Team Lead runs `dead_file_scan` before each phase kickoff
- **Phase completion check:** Phase is not marked **Complete** until `dead_file_scan` returns `high_confidence_count == 0`
- **Agent responsibility:** Any Implementation Agent that creates a file that supersedes another must delete the old file in the same commit
- **Systemic Critic gate:** If a Systemic Critic review finds orphaned files, it issues a `BLOCK` signal; work does not proceed until resolved

---

## Rationale

Files without context create three classes of harm:

1. **Cognitive load** — future agents read stale content and derive wrong conclusions
2. **Reference drift** — tools that scan the repo (capability maps, parity scanners, doc indexes) produce inflated or incorrect counts
3. **Trust erosion** — if the repo contains files that are demonstrably wrong or superseded, agents cannot trust the accuracy of any file

Keeping the file set minimal and intentional is a quality gate, not a housekeeping preference.
